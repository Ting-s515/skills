# 功能流程：訂單狀態更新後通知信寄送

## 功能概述

訂單狀態發生變更時，系統應可靠地觸發通知信並正確寄送給對應買家。

## 流程說明

### 比對條件

- 訂單 ID（`order_id`）
- 目標買家 Email（`buyer_email`）
- 新訂單狀態（`new_status`）
- 舊訂單狀態（`previous_status`）

### 處理流程

1. 後台管理員或系統更新訂單狀態（`new_status`）

2. 判斷狀態是否實際發生變更：
   - **`new_status` 與 `previous_status` 相同：** 不觸發通知，結束流程
   - **`new_status` 與 `previous_status` 不同：** 繼續往下執行

3. 查詢 `orders` 資料表，取得對應訂單資料：
   - 若訂單不存在（找不到 `order_id`）：記錄錯誤日誌，結束流程
   - 若訂單存在：繼續往下執行

4. 取得買家 Email（`buyer_email`）：
   - 若 `buyer_email` 為 `NULL` 或空字串：記錄警告日誌，跳過通知，結束流程
   - 若 `buyer_email` 有效：繼續往下執行

5. 判斷 `new_status` 是否屬於「需通知的狀態清單」（如：已出貨、已完成、已取消）：
   - **不在清單內：** 不觸發通知，結束流程
   - **在清單內：** 繼續往下執行

6. 依照 `new_status` 選擇對應的通知信模板

7. 將通知任務推送至非同步佇列（`notification_queue`），包含：
   - `order_id`、`buyer_email`、`new_status`、`template_id`

8. 佇列消費端從佇列取出任務，呼叫郵件服務（Email Service）發送通知信：
   - **郵件服務呼叫成功：** 更新通知發送狀態為「已發送」，記錄日誌
   - **郵件服務呼叫失敗（含 timeout）：** 進入重試機制（最多 3 次，指數退避）
     - **重試成功：** 更新通知發送狀態為「已發送」
     - **重試 3 次均失敗：** 更新通知發送狀態為「發送失敗」，記錄錯誤日誌，不再重試

9. 重複觸發保護：同一 `order_id` 同一 `new_status` 組合，若已有「已發送」紀錄，則跳過發送，避免重複通知
