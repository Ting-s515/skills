# 功能流程：修正訂單狀態通知信觸發流程

## 功能概述

確保訂單狀態更新時，系統正確觸發通知信並寄送給買家。

## 流程說明

### 比對條件

- 訂單狀態發生變更（`order.status` 由舊狀態轉為新狀態）
- 買家具備有效的 `email` 欄位（非空、非 NULL）

### 處理流程

1. 訂單狀態更新操作執行完成後，系統觸發通知事件

2. 查詢該訂單對應的買家資料，取得 `buyer_email`、`buyer_name`、`order_id`、`new_status`

3. **判斷買家 email 是否有效：**
   - **`buyer_email` 為 NULL 或空字串：** 記錄警告日誌，標記通知失敗，結束流程（不寄送）
   - **`buyer_email` 有效：** 繼續往下執行

4. **判斷新狀態是否屬於需通知的狀態清單：**
   - **不在通知清單內（如內部流程狀態）：** 跳過通知，結束流程
   - **在通知清單內（如：已確認、已出貨、已完成、已取消）：** 繼續往下執行

5. 依照 `new_status` 選取對應的通知信模板

6. 組合通知信內容，填入 `buyer_name`、`order_id`、`new_status` 對應的說明文字

7. 呼叫郵件服務 API 發送通知信

8. **判斷郵件服務回應：**
   - **發送成功：** 記錄成功日誌，更新訂單的 `notification_sent_at` 時間戳，結束流程
   - **發送失敗（API 錯誤或逾時）：** 記錄錯誤日誌，將通知任務加入重試佇列（最多重試 3 次），結束本次流程

9. **重試佇列處理（非同步）：**
   - 每次重試間隔遞增（第 1 次 1 分鐘、第 2 次 5 分鐘、第 3 次 15 分鐘）
   - 重試成功：更新 `notification_sent_at`，移出佇列
   - 超過最大重試次數：標記通知永久失敗，記錄警告日誌，通知管理員

### 邊界條件

- 同一訂單狀態若在短時間內觸發多次（重複更新），需確認冪等保護，避免重複寄送通知信
- 若買家 email 欄位有效但格式不正確，應在組合信件前做格式驗證，驗證失敗記錄錯誤並跳過
- 郵件服務完全不可用時，重試機制確保最終一致性；若仍失敗，管理員需手動處理
