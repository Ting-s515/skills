Code Review：Login API 實作
對照規格文檔：docs/specs/login-spec.md
審查對象：src/login.ts（git diff --staged）
審查時間：2026-05-17

==============================================================
整體摘要
==============================================================

本次實作從 stub（回傳 'not implemented'）升級為完整的登入處理邏輯，
整體方向正確，但有多處與規格不符的嚴重問題，需在合入前修正。

==============================================================
問題清單
==============================================================

【嚴重】問題 1：使用者不存在時回傳 404，違反安全規範

規格要求（安全規範第一點）：
  「使用者不存在與密碼錯誤必須回傳相同的錯誤訊息與狀態碼，
    避免帳號枚舉攻擊（Account Enumeration）」

實作：
  if (!user) {
    // 規格要求 401，且不可揭露使用者是否存在（避免帳號枚舉攻擊）
    return res.status(404).json({ error: '使用者不存在' });
  }

問題：
  1. 狀態碼用了 404，規格明確要求回傳 401
  2. 錯誤訊息「使用者不存在」直接揭露帳號是否存在，
     與「避免帳號枚舉攻擊」的安全規範完全相反
  3. 程式碼注釋寫著「規格要求 401」，但實際 status code 是 404，
     注釋與實作不一致，顯示開發者有意識到此要求卻未正確實作

修正方向：
  if (!user) {
    return res.status(401).json({ error: 'email 或密碼不正確' });
  }

--------------------------------------------------------------

【嚴重】問題 2：密碼錯誤時回傳 400 且洩漏錯誤原因

規格要求：
  - 密碼錯誤應回傳 401
  - 錯誤訊息應與「使用者不存在」的情況相同，不可洩漏差異

實作：
  if (!isValid) {
    // 規格要求 401
    return res.status(400).json({ error: '密碼錯誤' });
  }

問題：
  1. 狀態碼用了 400，規格要求 401
  2. 錯誤訊息「密碼錯誤」揭露了密碼本身是錯的（帳號存在），
     攻擊者可透過此差異推斷帳號是否存在
  3. 同樣，注釋寫「規格要求 401」但 status 是 400

修正方向：
  if (!isValid) {
    return res.status(401).json({ error: 'email 或密碼不正確' });
  }

--------------------------------------------------------------

【中】問題 3：缺少對 email / password 型別的驗證

規格欄位型別要求：
  - email: string
  - password: string

目前的驗證：
  if (!email || !password) {
    return res.status(422).json({ error: '缺少必填欄位' });
  }

此驗證僅確認欄位是否存在（truthy check），無法阻擋以下情況：
  - 傳入數字、物件、陣列等非字串型別
  - 空白字串（'' 是 falsy，會被正確捔住，但 ' ' 不會）

建議加上型別確認：
  if (typeof email !== 'string' || typeof password !== 'string' || !email.trim() || !password.trim()) {
    return res.status(422).json({ error: '缺少必填欄位' });
  }

--------------------------------------------------------------

【低】問題 4：JWT_SECRET 未做 undefined 保護

實作：
  const token = jwt.sign(
    { userId: user.id },
    process.env.JWT_SECRET as string,
    { expiresIn: '7d' }
  );

使用 `as string` 強制型別轉換，若 JWT_SECRET 未設定，
jwt.sign 實際上會收到 `undefined`，在某些版本下可能靜默失敗或產生無效 token。

建議在 handler 最前方或啟動時加上環境變數守衛：
  if (!process.env.JWT_SECRET) {
    throw new Error('JWT_SECRET environment variable is not set');
  }

--------------------------------------------------------------

【通過】符合規格的項目

1. 請求驗證：缺少 email 或 password 時回傳 422，符合規格
2. JWT 過期時間：expiresIn: '7d'，符合規格「7 天」
3. JWT 密鑰來源：從 process.env.JWT_SECRET 讀取，符合「不可硬編碼」規範
4. 成功回應格式：{ token, user: { id, email } }，符合 200 回應規格
5. 密碼比對：使用 bcrypt.compare 進行安全比對

==============================================================
修正優先順序
==============================================================

P0（必須修正，涉及安全漏洞）：
  - 問題 1：使用者不存在 → 改為 401 + 統一錯誤訊息
  - 問題 2：密碼錯誤 → 改為 401 + 統一錯誤訊息

P1（建議修正）：
  - 問題 3：加強輸入型別驗證

P2（可視情況處理）：
  - 問題 4：加上 JWT_SECRET undefined 保護

==============================================================
結論
==============================================================

此次實作在登入流程的整體架構上是正確的（使用 bcrypt、JWT、
正確處理必填欄位），但安全規範的兩個核心要求均未達成：

  1. 使用者不存在與密碼錯誤的狀態碼不同（404 vs 400），規格要求兩者都是 401
  2. 錯誤訊息洩漏了使用者是否存在的資訊

這兩個問題均違反 Account Enumeration 防護規範，會導致安全漏洞，
建議修正後再合入主線。
