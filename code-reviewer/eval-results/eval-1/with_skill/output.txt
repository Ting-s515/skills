## 📋 Code Review 摘要

**審查範圍：** `src/login.ts` — Login API 實作（POST /api/auth/login）
**整體評估：** ❌ 不符合規格，需修正

---

### 📐 規格符合度（有提供規格文檔時才輸出此區塊）

#### ✅ 符合規格的項目
- **422 驗證錯誤**：缺少必填欄位時正確回傳 `422` 狀態碼
- **JWT 過期時間**：`expiresIn: '7d'` 符合規格要求的 7 天
- **JWT 密鑰來源**：從 `process.env.JWT_SECRET` 讀取，未硬編碼
- **成功回應結構**：回傳 `{ token, user: { id, email } }` 符合規格定義

#### ❌ 不符合或缺漏的項目
- **帳號枚舉攻擊防護（Account Enumeration）**：規格明確要求使用者不存在與密碼錯誤必須回傳**相同的狀態碼與錯誤訊息**，但實作分別回傳 `404` / `使用者不存在` 與 `400` / `密碼錯誤`，完全洩漏帳號是否存在
- **使用者不存在的狀態碼**：規格要求回傳 `401`，實作卻回傳 `404`
- **密碼錯誤的狀態碼**：規格要求回傳 `401`，實作卻回傳 `400`

---

### 🔴 必須修正（Critical）

#### 問題 1：帳號枚舉攻擊防護完全未實作

- **檔案：** `src/login.ts:14-18`
- **問題：** 使用者不存在時回傳 `404` + `'使用者不存在'`；密碼錯誤時回傳 `400` + `'密碼錯誤'`。攻擊者可透過不同的狀態碼與訊息判斷帳號是否存在於系統中，這與規格中「避免帳號枚舉攻擊」的安全規範直接牴觸。
- **影響：** 惡意使用者可批次探測帳號是否存在，增加帳號暴力破解、社工攻擊的風險。
- **建議修正：**
  ```typescript
  const user = await User.findOne({ where: { email } });
  if (!user) {
    // 回傳與密碼錯誤相同的狀態碼與訊息，避免帳號枚舉攻擊
    return res.status(401).json({ error: 'email 或密碼不正確' });
  }

  const isValid = await bcrypt.compare(password, user.passwordHash);
  if (!isValid) {
    return res.status(401).json({ error: 'email 或密碼不正確' });
  }
  ```

### 🟠 建議改善（Warning）

#### 問題 1：JWT_SECRET 未設定時的 runtime 風險

- **檔案：** `src/login.ts:22`
- **問題：** `process.env.JWT_SECRET as string` 使用 TypeScript 型別斷言，若環境變數未設定，`jwt.sign` 會收到 `undefined`，在 runtime 可能拋出例外或產生使用 `undefined` 作為密鑰的 JWT，造成嚴重安全漏洞。
- **影響：** 在未正確設定環境變數的部署環境中，可能靜默產生不安全 token，或導致服務崩潰。
- **建議修正：**
  ```typescript
  const secret = process.env.JWT_SECRET;
  if (!secret) {
    // 啟動時或處理請求時提早失敗，避免靜默安全漏洞
    throw new Error('JWT_SECRET 環境變數未設定');
  }
  const token = jwt.sign({ userId: user.id }, secret, { expiresIn: '7d' });
  ```

### ⚪ 使用者自行決定（註解類問題）

#### 問題 1：程式碼內的規格參照註解與實作不一致

- **檔案：** `src/login.ts:15` 與 `src/login.ts:19`
- **問題：** 兩處註解皆寫「規格要求 401」，但實際回傳的分別是 `404` 與 `400`，造成程式碼意圖與實作明顯矛盾。即使此為開發中狀態，這類不一致的技術債務會使後續維護者難以判斷哪個才是正確行為。
- **影響：** 若進入正式環境仍保留此狀態，可能引發混淆並阻礙日後的安全性審查。
- **建議：** 修正實作使其符合規格（同問題 1 建議），並同步移除或更新相關說明性註解。
