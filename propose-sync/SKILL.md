# Propose Sync

掃描 `docs/propose/` 所有功能資料夾的任務完成狀態，將已完成的功能同步回規格需求文檔。

## 執行流程

### Step 1：確認規格文檔路徑與根路徑前綴

詢問使用者：

```
請提供要同步的規格需求文檔路徑：
（例如：docs/spec.md）
```

同時從使用者指令或規格文檔路徑判斷根路徑前綴：

- 使用者指定「前端」或「frontend」→ 掃描 `frontend/docs/propose/`
- 使用者指定「後端」或「backend」→ 掃描 `backend/docs/propose/`
- 未指定 → **主動詢問**：「請問要同步的是前端還是後端，或是不區分？」，依回答決定掃描目錄（不區分則掃描 `docs/propose/`）

### Step 2：掃描 propose 目錄

讀取對應目錄下所有子資料夾的 `03-tasks.md`，判斷完成狀態：

**完成判斷規則：**

- 所有任務列表項目（`- [ ]` 或 `- [x]`）全部符合 `[x][cr]` 格式 → 視為已完成
- 有任何任務為 `[ ]` 或僅有 `[x]` 未有 `[cr]` → 視為未完成
- `03-tasks.md` 不存在 → 視為未完成

掃描完成後**直接進入 Step 3 更新規格文檔**，無需等待使用者確認。

### Step 3：更新規格文檔

讀取規格文檔，更新最上方的 `## 已完成` 區塊：

- 若 `## 已完成` 區塊**不存在**，在文檔最頂部插入
- 若已存在，**完整替換**區塊內容（避免重複累積）

格式：

```markdown
## 已完成

| 功能       | 資料夾                               |
| ---------- | ------------------------------------ |
| <功能描述> | `<掃描路徑>/feature-a/` |
| <功能描述> | `<掃描路徑>/feature-b/` |

---
```

**功能描述**從規格文檔中對應的 `> propose:` 標記上方段落取得；若找不到對應標記，使用 folder name 作為描述。

### Step 4：輸出結果

```
propose-sync 完成：

同步至：<規格文檔路徑>

已新增至 ## 已完成：
  - feature-a
  - feature-b

未完成功能不受影響，可繼續使用 apply skill 實作。
```
