# Keycloak 在 .NET 後端的 JWT 驗證與 Middleware 設定

根據 `[[keycloak-backend]]` 與 `[[keycloak-integration]]`：

## 整體認證架構（wiki 流程圖）
收到 API Request
  → 健康檢查路徑（/_hc）→ 直接放行
  → [SkipCurrentUser] → 放行
  → Header 含 apiKey → 測試模式（僅 Development）
  → JWT Bearer 驗證（JWKS 簽章 + Claims 解析）

## JWT Token 驗證參數
ValidateIssuer: true
ValidateAudience: true
ValidateLifetime: true
ClockSkew: TimeSpan.Zero（嚴格過期驗證）
簽章驗證：JWKS（Keycloak 公鑰端點）

## Middleware 執行順序
app.UseAuthentication();       // 1. 驗證 JWT Bearer Token
app.UseAuthorization();        // 2. 授權檢查
// RequestBodyCaptureMiddleware // 3. 擷取 Body
// TrackIdMiddleware            // 4. 追蹤 ID
// CurrentUserMiddleware        // 5. 解析 Claims → 使用者物件

## JWT Claim 對照表（Keycloak 欄位）
sub → User ID (GUID)
preferred_username → 登入帳號
email → 電子郵件
realm_access.roles → 角色清單
unit / phone → 自訂屬性（需 Mapper）

## MultiScheme 雙認證
正式：JWT Bearer（Keycloak）
開發：apiKey header → 固定測試用戶

# Rust 的 Ownership 機制

## 三條核心規則
1. 每個值都有且只有一個 Owner
2. Owner 離開作用域時，值自動被釋放（drop）
3. 所有權可以「移動」（Move），一次只能有一個 Owner

## Move / Clone / Copy 語義說明（含程式碼範例）

## Borrowing（& 不可變 / &mut 可變借用）
- 同一時刻只能有一個可變借用
- 可變借用與不可變借用不能同時存在
