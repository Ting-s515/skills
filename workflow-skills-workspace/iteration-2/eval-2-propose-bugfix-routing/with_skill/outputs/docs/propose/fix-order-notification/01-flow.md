# 功能流程：訂單狀態更新通知信觸發修正

## 功能概述

確保訂單狀態更新後，系統能正確觸發並寄送通知信給對應買家。

## 流程說明

### 比對條件

- `訂單 ID`：用於查詢訂單與買家資訊
- `訂單狀態`：觸發通知的關鍵欄位，不同狀態對應不同通知內容

### 處理流程

1. 訂單狀態發生變更（由後台操作員或系統自動觸發）

2. 系統查詢對應訂單記錄：
   - **訂單不存在：** 記錄錯誤 log，結束流程（Out of Scope：不對外回傳錯誤）
   - **訂單存在：** 繼續往下執行

3. 取得買家 Email：
   - **Email 為 NULL 或空字串：** 記錄 warning log，跳過通知，結束流程
   - **Email 有效：** 繼續往下執行

4. 依據新訂單狀態決定通知信模板：
   - **狀態為「已確認」：** 使用「訂單確認」模板
   - **狀態為「已出貨」：** 使用「出貨通知」模板
   - **狀態為「已完成」：** 使用「交易完成」模板
   - **狀態為「已取消」：** 使用「取消確認」模板
   - **其他狀態：** 不觸發通知，結束流程

5. 呼叫郵件發送服務（Email Service）：
   - **發送成功：** 更新訂單通知記錄（`notification_sent = true`，記錄發送時間）
   - **發送失敗（服務錯誤或逾時）：**
     - 記錄錯誤 log（含訂單 ID、狀態、錯誤原因）
     - 將通知任務加入重試佇列（最多重試 3 次）
     - **重試 3 次均失敗：** 標記通知為「發送失敗」，發出告警通知給系統管理員

6. 流程結束

### 邊界條件

| 情境 | 處理方式 |
|------|----------|
| 訂單不存在 | 記錄 error log，不拋出例外 |
| 買家 Email 為空 | 記錄 warning log，跳過通知 |
| Email Service 不可用 | 進入重試佇列，最多 3 次 |
| 重複觸發相同狀態更新 | 依 `notification_sent` 欄位防止重複發送 |
| 不支援通知的狀態 | 靜默跳過，不記錄錯誤 |
