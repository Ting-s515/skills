# 功能流程：商品庫存歸零時前台購買按鈕狀態同步修正

## 功能概述

確保商品庫存歸零時，前台「可購買」按鈕即時更新為禁用狀態，防止買家對零庫存商品下單。

## 流程說明

### 比對條件

- `商品 ID`：用於查詢庫存數量
- `庫存數量`：判斷按鈕狀態的關鍵欄位

### 處理流程

#### A. 頁面初始載入

1. 前台商品頁面載入時，呼叫商品詳情 API 取得最新庫存數量

2. 依據庫存數量決定按鈕狀態：
   - **庫存 > 0：** 顯示「加入購物車」或「立即購買」按鈕（enabled）
   - **庫存 = 0：** 顯示「已售完」或「補貨中」文字，按鈕設為 disabled（不可點擊）

#### B. 庫存即時更新（庫存異動後同步前台狀態）

1. 後台操作員更新商品庫存，或系統因訂單成立自動扣減庫存

2. 庫存更新完成後，觸發庫存異動事件（`stock_updated` event）

3. 系統推播庫存狀態變更通知至前台（可選 WebSocket 或 polling 機制，依現有架構決定）：
   - **庫存從 > 0 變為 0：** 前台按鈕切換為 disabled
   - **庫存從 0 恢復為 > 0：** 前台按鈕切換為 enabled
   - **庫存在 > 0 的範圍內變動：** 不影響按鈕狀態，無需更新 UI

4. 前台接收到狀態變更後更新按鈕狀態

#### C. 邊界條件

| 情境 | 處理方式 |
|------|----------|
| 商品不存在 | API 回傳 404，前台顯示商品不存在提示 |
| 庫存欄位為 NULL | 視為庫存不明，按鈕保持 disabled（保守策略） |
| 即時推播失敗 | 前台 polling 作為降級機制，每 x 秒查詢一次 |
| 使用者停留在頁面期間庫存歸零 | 使用者嘗試下單時，後端進行最終庫存驗證並返回錯誤 |
| 多個商品頁面同時開啟 | 各頁面獨立查詢，不共享狀態快取 |
