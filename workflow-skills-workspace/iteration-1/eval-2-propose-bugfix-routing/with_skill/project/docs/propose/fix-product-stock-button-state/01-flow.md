# 功能流程：修正商品庫存歸零時購買按鈕狀態

## 功能概述

當商品庫存歸零時，前台商品頁面的購買按鈕應正確顯示為不可購買狀態，防止使用者誤操作。

## 流程說明

### 比對條件

- 商品的 `stock_quantity` 欄位值為 0 或小於 0
- 前台商品頁面載入或庫存資料更新時觸發按鈕狀態判斷

### 處理流程

1. 前台商品頁面載入時，向後端 API 請求商品資料，回應包含 `product_id`、`name`、`price`、`stock_quantity`

2. **判斷 `stock_quantity` 是否大於 0：**
   - **`stock_quantity > 0`（有庫存）：** 將購買按鈕設為啟用狀態，顯示文字「加入購物車」或「立即購買」
   - **`stock_quantity <= 0`（無庫存）：** 將購買按鈕設為停用狀態（disabled），顯示文字「已售完」，阻擋點擊事件

3. **庫存資料即時更新情境（如 WebSocket 或輪詢）：**
   - 接收到庫存更新事件後，重新判斷 `stock_quantity`
   - 依上述判斷規則更新按鈕狀態，不需整頁重整

4. **使用者嘗試點擊停用按鈕（防呆保護）：**
   - 前端阻擋提交動作
   - **若前端判斷失效，後端 API 也應驗證庫存：**
     - 庫存不足時回傳錯誤，前台顯示提示訊息「此商品已售完，無法購買」

### 邊界條件

- `stock_quantity` 為 NULL 時，視為無庫存（停用按鈕），不假設有庫存
- 庫存數量為負數時，同樣視為無庫存（停用按鈕）
- 多個使用者同時購買最後一件商品時，後端需確保庫存扣減的原子性，前端按鈕狀態在購買完成後應反映最新庫存
- 商品上架/下架狀態與庫存狀態獨立判斷，下架商品無論庫存多少皆不可購買（Out of Scope：下架邏輯由其他功能管理）
