# 優化前後對照範例（複雜嵌套流程）

本範例以「訂單審核流程」為情境，展示多層判斷分支、迴圈遍歷、提前退出的結構化寫法，
適用於任何業務流程（審核、資格驗證、通知派送等）。

---

## 優化前（口語化、結構鬆散）

```
拿到申請單後，先看有沒有過期，過期就拒絕。
沒過期的話去看審核人清單，一個一個確認有沒有人批准，
有批准的話再看金額，超過授權額度就要升級審核，
沒超過就直接過，找不到審核人就退回。
```

---

## 優化後（結構化、可對應 Mermaid 流程圖）

**觸發條件：** 收到申請單（包含申請單 ID、申請金額、提交時間）

**處理流程：**

1. 檢查申請單的有效期限（`ExpiresAt`）：
   - **已過期（`ExpiresAt` < 當前時間）：** 將申請單狀態更新為「已拒絕」，結束流程
   - **未過期：** 承接第 2 點

2. 取得該申請單對應的審核人清單（依 `ReviewerOrder` 欄位由小到大排序）

3. 依序遍歷審核人清單，逐一判斷每位審核人的審核狀態（`ReviewStatus`）：

   - **狀態為「待審」：** 跳過，繼續遍歷下一位
   - **狀態為「已拒絕」：** 將申請單標記為「審核不通過」，結束遍歷，結束流程
   - **狀態為「已批准」：** 承接第 4 點，進行金額授權判斷，結束遍歷
   - **遍歷完畢，無任何「已批准」：** 將申請單狀態更新為「待審核」，結束流程（等待下一輪觸發）

4. 判斷申請金額（`Amount`）是否超過審核人授權額度（`ApprovalLimit`）：

   - **未超過（`Amount` <= `ApprovalLimit`）：** 將申請單狀態更新為「已核准」，結束流程
   - **超過（`Amount` > `ApprovalLimit`）：** 建立升級審核任務，指派給上層主管，結束流程

---

## 對照說明

| 優化原則 | 對應位置 |
|----------|----------|
| 明確資料來源 | 步驟 2：指明查詢審核人清單，並說明排序依據 |
| 列舉篩選條件 | 步驟 3：逐一列出每個審核狀態的對應行為 |
| Null 值 / 空清單處理 | 步驟 3：「遍歷完畢，無任何已批准」的結束點 |
| 粗體標記分支 | 步驟 3-4：**狀態為「已拒絕」**、**未超過**、**超過** |
| 縮排呈現層級 | 步驟 3：子判斷以縮排清單表示 |
| 明確遍歷對象 | 步驟 3：「審核人清單」 |
| 提前退出 | 步驟 3：找到「已拒絕」或「已批准」即結束遍歷 |
| 遍歷完畢處理 | 步驟 3：全部為「待審」時的 fallback 行為 |
| 步驟連貫追蹤 | 步驟 1 → 步驟 3 → 步驟 4：以「承接第 N 點」串接 |
